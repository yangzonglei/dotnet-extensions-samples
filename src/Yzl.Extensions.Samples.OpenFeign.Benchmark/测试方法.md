# OpenFeign 性能对比测试方法

## 测试目的

对比 `Yzl.Extensions.Http.OpenFeign` 三个版本的性能：
- **0.1.18** — 原始版本
- **0.1.19** — 原始版本
- **修改版** — 本地源码（MaxConnectionsPerServer + HandleSyncResult 超时 + 异常分类）

## 测试环境

| 项目 | 说明 |
|:----|:------|
| 操作系统 | macOS / Linux |
| CPU | 10 核（实测） |
| 内存 | 16G |
| .NET 版本 | 8.0 |
| ThreadPool | min(10,1) max(32767,1000) |

## Feign 测试配置

所有版本使用相同配置（硬编码在 `Program.cs` 的 `BuildFeignClient()` 方法中）：

```yaml
spring:
  feign:
    default:
      timeout: 3000
      syncTimeoutMs: 15000
      maxConcurrentRequests: 0
      retry:
        enabled: true
        maxAttempts: 2
        delayMs: 500
        maxDelay: 2000
    httpClient:
      handlerLifetime: "00:30:00"
      pool:
        maxConnectionsPerServer: 300
```

> 注意：原始版本（0.1.18 / 0.1.19）不识别 `syncTimeoutMs`、`maxConcurrentRequests`、`MaxConnectionsPerServer` 配置项，这些配置仅在修改版中生效。

## 测试场景

### 场景 1：高频短请求

模拟最常见的 API 调用场景：请求处理快、返回数据小。

```
目标: /api/ping（返回 "pong"）
请求数: 3000
并发度: 300
预期: 观察连接池复用效率和同步方法线程阻塞程度
```

### 场景 2：混合请求

模拟真实业务中不同接口的混合调用。

```
目标: /api/ping + /api/users/{id}
请求数: 3000（20% GetUser + 80% Ping）
并发度: 300
预期: 观察多端点下的连接池行为
```

### 场景 3：超时压力

模拟下游服务变慢时，Feign 客户端的超时重试和回退行为。

```
目标: /api/timeout（内部 Thread.Sleep(6000)）
请求数: 200
并发度: 100
预期: 观察超时重试对线程的阻塞情况和修改版的兜底超时效果
```

### 场景 4：短暂突发

模拟流量突增时的高并发处理能力。

```
目标: /api/ping
请求数: 500
并发度: 500
预期: 观察突发高并发下的吞吐和延迟
```

## 测试步骤

### 准备工作

```bash
# 1. 确认 dotnet-extensions 和 dotnet-extensions-samples 在同一级目录
ls ../dotnet-extensions   # 主仓库（修改版源码在这里）
ls .                      # 当前在 dotnet-extensions-samples
```

### 第一步：测试修改版（本地源码）

```bash
cd src/Yzl.Extensions.Samples.OpenFeign.Benchmark

dotnet run -p:UseProjectReference=true | tee /tmp/benchmark-modified.log
```

耗时约：**2 分钟**

### 第二步：测试 0.1.19（NuGet）

```bash
# 修改 samples 仓库的 Directory.Packages.props
# 将 Yzl.Extensions.Http.OpenFeign 的 Version 改为 0.1.19

dotnet run | tee /tmp/benchmark-019.log
```

耗时约：**2 分钟**

### 第三步：测试 0.1.18（NuGet）

```bash
# 修改 samples 仓库的 Directory.Packages.props
# 将 Yzl.Extensions.Http.OpenFeign 的 Version 改为 0.1.18

dotnet run | tee /tmp/benchmark-018.log
```

耗时约：**2 分钟**

### 第四步：对比结果

```bash
echo "=== 0.1.18 ===" && \
```

## 关注指标

| 指标 | 说明 | 优劣判断 |
|:----|:-----|:--------|
| **错误数** | 总失败请求数 | 越低越好，应全为 0 |
| **吞吐 (req/s)** | 每秒完成的请求数 | 越高越好 |
| **P50 延迟** | 50% 请求在此延迟以下 | 越低越好 |
| **P95 延迟** | 95% 请求在此延迟以下 | 越低越好，关注高水位 |
| **P99 延迟** | 99% 请求在此延迟以下 | 越低越好，关注极端情况 |
| **ThreadPool 峰值** | 同时占用的工作线程数 | 越低越好，说明线程效率高 |
| **错误类型分布** | 如有时，区分超时/网络/反序列化 | 修改版应分类更清晰 |

## 预期差异

| 场景 | 0.1.18 vs 0.1.19 | 修改版 vs 原始版 |
|:----|:----------------:|:----------------:|
| 高频短请求 | 应无明显差异 | 应无明显差异（同步瓶颈相同） |
| 混合请求 | 应无明显差异 | 应无明显差异（同步瓶颈相同） |
| 超时压力 | 应无明显差异 | **P99 应降低 30-50%**（HandleSyncResult 兜底超时） |
| 短暂突发 | 应无明显差异 | **吞吐应提升 2-3x**（MaxConnectionsPerServer 连接池优化） |
| 错误报告 | 同为 "Failed to deserialize" 模糊消息 | **异常分类清晰**（超时/网络/Socket/JSON 分别提示） |

## 注意事项

1. **顺序执行的影响**：三个版本的测试在同一个机器上顺序执行，后跑的版本受之前测试残留负载的影响，数据会有噪声。建议每次跑完等待 30s 再跑下一个。

2. **同步瓶颈不可消除**：同步 Feign 接口方法（非 `Task<T>`）会阻塞线程池线程，这是架构级约束，代码层面只能缓解不能消除。预期三版本在高并发 Ping 场景下吞吐相近。

3. **线程池配置影响**：默认 `ThreadPool.SetMinThreads(10, 1)` 在高并发同步阻塞场景下是瓶颈。如需更高吞吐，可调大 MinThreads 后重新测试。

4. **版本标签显示**：程序输出中的版本标签基于编译符号 `USING_NUGET` / `USING_LOCAL`，**不能反映具体 NuGet 版本号**。使用时手动记录 `Directory.Packages.props` 中设置的版本。
